# PRD V3.1：《史隙》AI 驱动历史悬疑分支叙事游戏

## 一句话总结

**《史隙》是一款 AI 驱动的历史悬疑分支叙事游戏。玩家进入由 AI 构建的朝代事件世界，在视觉小说式场景中调查关键线索、与智能 NPC 对话，并通过每一次选择触发故事转折和多结局。系统以朝代知识库保证历史可信，以 AI 生成世界背景、人物关系、NPC 反馈、完整剧本包和关键视觉，以状态机、线索系统、一致性监管器、图片质量门禁和热点定位校准确保强故事线不失控。**

本项目不是普通历史聊天机器人，也不是单纯的历史知识问答。它的核心是：

> **用 AI 重塑剧情驱动类游戏体验：AI 构建世界观背景，AI 生成可玩的历史悬疑剧本包和关键视觉，AI NPC 推进剧情，玩家选择触发故事转折，最终形成分支叙事与多结局体验。**

# 1. 产品基本信息

## 1.1 产品名称

正式推荐名：**《史隙》**

含义：玩家不是改写整个历史，而是在历史缝隙中介入一个具体事件，看见时代结构如何挤压个人命运。

备选名：

| 名称     | 气质                |
| ------ | ----------------- |
| 《史隙》   | 简洁，有历史缝隙感，最适合当前方向 |

| 《入史者》  | 更强调玩家身份代入         |
| 《一念成史》 | 更商业化，强调选择改变命运     |
| 《逆史录》  | 冲突感强，但略中二         |

最终建议：**使用《史隙》**。

## 1.2 产品定位

**一款 AI 剧情驱动的历史悬疑分支叙事网页游戏。**

玩家每一局进入一个由 AI 构建的历史事件世界，例如：

* 明代书坊焚稿案；
* 北宋驿站军报案；
* 晚唐粮仓亏空案。

系统根据朝代知识库、事件模板、玩家身份和历史矛盾生成本局事件世界，包括：

* 世界观背景；
* 事件前史；
* 当前场景；
* 关键 NPC；
* 人物关系网；
* 可调查线索；
* 关键视觉；
* 分支剧情；
* 多结局条件。

玩家通过调查场景、点击红色高亮线索、查看线索栏、与 NPC 对话、出示证据和做出选择推动剧情。AI NPC 会根据玩家推理进度、已发现线索、信任度和自身知识权限动态回应，逐步释放信息。

# 2. 产品目标

## 2.1 核心目标

构建一个可运行、可演示、可重玩的网页端 AI 历史悬疑分支叙事游戏原型。

本项目的核心目标不是证明“AI 能写历史故事”，而是证明：

> **AI 可以在强故事线、世界观背景、人物关系、关键视觉、NPC 权限、线索系统和状态机约束下，生成并推动一局接近推理游戏 / 剧本杀完成度的历史悬疑事件。**

核心目标分为四层。

---

## 2.2 第一层：游戏体验目标

玩家每一局都应获得一个完整、有代入感、有选择压力的历史悬疑故事，而不是几段松散 AI 对话。

| 目标     | 具体要求                                  |
| ------ | ------------------------------------- |
| 强故事线   | 每局必须有开场钩子、嫌疑关系、线索链、误导线、中段反转、关键抉择和结局回收 |
| 推理成立   | 真相不能靠 NPC 直接说出，必须由玩家通过线索逐步拼出          |
| NPC 可信 | NPC 必须有明确动机、隐瞒理由、情感压力和信息边界            |
| 冲突有层次  | 事件不能只是“谁偷了东西”，而要连接朝代制度、权力结构、生计压力或身份困境 |
| 选择有代价  | 不做简单好坏结局，而是在真相、秩序、自保、他人命运之间做取舍        |
| 视觉有记忆点 | 主视觉、关键场景、NPC 剪影和线索插图必须服务剧情氛围          |
| 可演示    | 评委 3 分钟内能理解：AI 如何构建世界、推动剧情、释放线索、触发分支  |

---

## 2.3 第二层：AI 生成目标

AI 不负责随便编剧情，而是在受控框架中承担“剧情表现、世界细化和动态反馈”的角色。

| AI 能力     | 具体要求                            |
| --------- | ------------------------------- |
| 世界背景生成    | 根据朝代、事件模板、玩家身份生成本局事件世界          |
| 人物关系生成    | 生成 NPC 之间的利益关系、隐瞒关系、误判关系和冲突关系   |
| NPC 动态对话  | NPC 根据玩家输入、已发现线索、信任度和自身权限生成不同回应 |
| 线索解释生成    | AI 对玩家发现的线索进行符合时代背景的解释          |
| 分支反馈生成    | 玩家选择后，AI 生成即时剧情反馈、NPC 情绪变化和场景变化 |
| 历史回声生成    | 结局后，AI 总结玩家选择与时代结构之间的关系         |
| 关键视觉提示词生成 | AI 为主视觉、场景、NPC、关键线索生成统一风格提示词    |

---

## 2.4 第三层：历史可信目标

历史材料不是装饰，而是 AI 世界构建、NPC 对话和剧情监管的底层依据。

| 目标     | 具体要求                            |
| ------ | ------------------------------- |
| 历史可信   | 使用结构化朝代知识库 + RAG 检索，避免 AI 胡编    |
| 材料分级   | 区分一手记录、官修史书、制度文献、笔记材料、现代研究整理    |
| 朝代差异明显 | 北宋、晚唐、明代的职业、称谓、制度、空间、矛盾必须不同     |
| 历史进入玩法 | 文书、官制、驿站、粮册、城门搜检等内容必须能成为线索或选择依据 |
| 可追溯    | 关键历史设定在知识库中有来源等级与来源说明，便于调试和路演   |

---

## 2.5 第四层：技术可控目标

AI 必须参与核心体验，但不能失控。

| 目标     | 具体要求                      |
| ------ | ------------------------- |
| 剧情可控   | 主线阶段、线索释放、结局条件由状态机和规则控制   |
| AI 可监管 | 所有 AI 输出经过一致性监管器检查        |
| 输出结构化  | AI 尽量输出 JSON，便于前后端处理      |
| 失败可回退  | AI 调用失败、超时或输出违规时，使用预设文本回退 |
| 日志可展示  | 保留 AI 调用日志，用于证明真实 AI 接入   |
| API 安全 | API Key 只放后端，不暴露给前端和公开仓库  |

# 4. 核心玩家体验

## 4.1 玩家体验描述

玩家进入《史隙》后，不是单纯选择一个朝代然后看 AI 讲故事，而是进入一局完整的历史悬疑事件。

每一局都像一场小型剧本杀或推理文字冒险，有明确案件、复杂人物关系、可追查线索链、误导信息、中段反转和带有代价的结局。

玩家体验路径分为两类。

明代固定 Demo 路径：

```text
选择明代
  ↓
直接进入书坊焚稿案
  ↓
按现有调查、对话、线索、抉择和结局流程游玩
```

北宋 / 晚唐 AI 生成路径：

```text
选择北宋 / 晚唐
  ↓
填写 1-8 个关键词
  ↓
查看真实生成流程图：剧本初稿、审稿、逻辑 QA、图片生成、质量门禁、热点定位
  ↓
等待期间轮播历史名人名句
  ↓
生成完成后查看剧本概览
  ↓
从生成剧本的 playable_identities 中选择身份
  ↓
进入生成剧本
```

通用游玩路径：

```text
进入朝代事件世界
  ↓
看到 AI 生成的主视觉与开场背景
  ↓
获得身份：书坊学徒 / 驿站文书 / 县衙小吏
  ↓
卷入一个看似普通但明显不对劲的事件
  ↓
调查场景，点击红色高亮线索
  ↓
线索进入右侧线索栏
  ↓
与 NPC 对话，普通追问 / 证据追问 / 压力追问
  ↓
NPC 根据玩家推理进度动态释放信息
  ↓
线索组合触发阶段推进
  ↓
发现表层解释背后的时代矛盾
  ↓
做出关键抉择
  ↓
进入多结局
  ↓
AI 生成历史回声，总结玩家改变了什么、没能改变什么
```

## 4.2 主 Demo 玩家体验示例

玩家进入明代，身份是书坊学徒。

开局不是直接告诉玩家“有人烧毁了涉及官府账册的文稿”，而是从一个表面合理的事故开始：

> 夜雨未停，书坊后院忽然起火。掌柜说是灯油翻倒，烧掉的只是几卷未刊诗文。锦衣卫校尉很快赶到，封住前后门，要求所有人不得离开。
> 但玩家发现，火起处并不在灯油架旁，灰烬中残留半枚带红印的纸角，刻工阿沈明明说自己整夜在刻坊，却避开了关于“三更之后”的问题。

可高亮线索示例：

```html
火起的位置不在灯油架旁，而在靠近旧书箱的墙角。
灰烬里露出半枚纸角，上面残留着
<span class="clue-highlight">辽东粮册</span>
四字。
```

玩家初期只能判断：

* 火灾可能不是意外；
* 烧掉的可能不是普通诗文；
* 掌柜、刻工、士子、锦衣卫之间至少有人隐瞒了信息。

随着调查推进，玩家逐步发现：

* 掌柜提前搬走过一箱书稿；
* 刻工听见有人说“那箱东西不能留到天亮”；
* 士子顾闻的诗稿中夹带过一份抄录；
* 锦衣卫校尉陆峥并不知道完整真相，只是奉命封锁；
* 真正危险的不是禁书本身，而是文稿中牵出的边地粮册与官府往来。

最终玩家不是简单选择“揭发 / 不揭发”，而是在几种不完美方案中选择：

| 选择      | 可能结果              |
| ------- | ----------------- |
| 交给锦衣卫   | 秩序上安全，但证据可能被上层截断  |
| 暗中保存残页  | 真相保留，但玩家和士子风险上升   |
| 帮掌柜销毁余稿 | 自保成功，但关键证据消失      |
| 让士子带走抄录 | 人可能逃走，但书坊会被清算     |
| 反查幕后命令  | 接近真相，但极易触发悲剧或隐藏结局 |

## 4.3 玩家体验验收标准

| 验收问题          | 合格标准                           |
| ------------- | ------------------------------ |
| 玩家是否知道自己要查什么？ | 开局 1 分钟内给出明确异常事件               |
| 玩家是否有推理空间？    | 真相不能由 NPC 直接讲出，必须通过 6–10 条线索拼出 |
| NPC 是否像真人？    | 每个 NPC 都有自己的利益、恐惧、隐瞒和信息边界      |
| AI 是否真正推动剧情？  | NPC 回复、线索解释、分支反馈和历史回声由 AI 动态生成 |
| 线索是否有用？       | 每条线索都影响对话、信任、阶段推进或结局           |
| 反转是否合理？       | 反转必须由前置线索支持，不能突然空降             |
| 结局是否回收前文？     | 结局文本必须引用玩家关键线索和关键选择            |
| 视觉是否服务剧情？     | 主视觉、场景图、NPC 图和线索图必须强化悬疑氛围      |
| 历史感是否进入玩法？    | 朝代制度、文书、官府权力、职业身份必须影响推理过程      |

# 5. 核心玩法设计

## 5.1 单局剧本结构

每局事件由七个部分组成：

```text
事件钩子
  ↓
表层解释
  ↓
异常细节
  ↓
嫌疑关系网
  ↓
线索链推进
  ↓
中段反转
  ↓
关键抉择与结局回收
```

对应游戏流程：

```text
进入首页
  ↓
选择朝代
  ↓
选择 / 随机身份
  ↓
AI + 规则生成本局事件世界
  ↓
生成世界背景、人物关系、场景、线索、视觉提示词
  ↓
进入第一场景
  ↓
玩家阅读开场事件与主视觉
  ↓
发现第一批红色高亮线索
  ↓
调查 / 对话 / 出示线索 / 做选择
  ↓
线索进入右侧线索栏
  ↓
AI NPC 根据状态和权限生成回复
  ↓
一致性监管器审查 AI 输出
  ↓
状态机更新剧情阶段、信任度、风险值和结局分数
  ↓
触发误导、矛盾证言或新场景
  ↓
中段反转
  ↓
玩家整理线索并做关键抉择
  ↓
结局判定
  ↓
AI 生成历史回声
```

---

## 5.2 玩家每轮操作

| 操作       | 说明                       |
| -------- | ------------------------ |
| 查看场景     | 阅读当前地点、人物、异常物品和可调查对象     |
| 调查对象     | 点击火场、残页、账册、脚印、锁痕、文书等     |
| 与 NPC 对话 | 使用推荐追问或有限自由输入            |
| 出示线索     | 将已获得线索拿给 NPC 质问或试探       |
| 压力追问     | 用矛盾证据逼迫 NPC 改变说法         |
| 整理线索     | 在线索栏中尝试组合证据，形成推理结论       |
| 做出选择     | 在关键节点选择隐瞒、上报、保护、交换、反查等行动 |

其中，“出示线索”是核心操作。否则线索栏只是收藏夹，不能真正参与推理。

---

## 5.3 系统每轮操作

```text
接收玩家行为
  ↓
读取 GameState
  ↓
判断行为类型：
    - 调查
    - 对话
    - 出示线索
    - 整理线索
    - 关键选择
  ↓
检索相关历史知识和本局剧本规则
  ↓
读取 NPC 角色卡、知识权限、信任度
  ↓
调用 AI 生成场景反馈或 NPC 回复
  ↓
一致性监管器检查：
    - 历史错误
    - 错朝代元素
    - 剧透
    - 人设漂移
    - 线索冲突
    - 阶段跳跃
  ↓
若失败则修复或回退预设文本
  ↓
更新：
    - 已发现线索
    - NPC 信任度
    - 剧情阶段
    - 风险值
    - 真相分数
    - 秩序分数
    - 自保分数
  ↓
返回给前端渲染
```

---

## 5.4 线索驱动剧情推进

剧情阶段不能只由“对话次数”推进，而应由线索和状态触发。

| 阶段变化                     | 推进条件                            |
| ------------------------ | ------------------------------- |
| intro → investigation    | 玩家发现至少 1 条异常线索                  |
| investigation → reversal | 玩家发现至少 3 条核心线索，或识破 1 个 NPC 矛盾证言 |
| reversal → choice        | 玩家完成至少 1 组线索组合，确认表层解释不成立        |
| choice → ending          | 玩家做出关键抉择                        |
| ending                   | 根据线索完整度、NPC 信任、风险值和选择判定结局       |

---

## 5.5 误导线设计

推理游戏不能只有一条直线真相。每局至少有一条误导线。

明代书坊焚稿案误导线：

| 误导线   | 表面解释           | 后续修正                  |
| ----- | -------------- | --------------------- |
| 禁书嫌疑  | 玩家以为书坊烧的是犯禁文稿  | 后续发现禁书只是掩护，真正危险的是粮册抄录 |
| 掌柜主谋  | 掌柜提前搬箱，看似主谋    | 后续发现他只是被迫自保，并不掌握完整黑幕  |
| 锦衣卫主谋 | 锦衣卫封锁现场，看似幕后黑手 | 后续发现陆峥只是执行命令，也被上级利用   |

## 6.4 DeepSeek 剧本包生成器

Stage 15 新增 DeepSeek 剧本包生成器。它不是“明代书坊案变体生成器”，而是北宋 / 晚唐入口的通用历史悬疑剧本生成器。当前兼容规则如下：

```text
明代：固定书坊焚稿案 Demo，不进入 AI 生成流程。
北宋：用户填写关键词后生成新剧本。
晚唐 / 唐后期：用户填写关键词后生成新剧本。
```

生成结果必须是结构化 `ScriptPackage`，至少包含：

- `script_overview`：非剧透剧本概览；
- `playable_identities`：生成身份列表；
- `world`、`story`、`stages`：世界状态和阶段结构；
- `locations`、`npcs`、`relationships`：地点、人物和关系；
- `clues`、`clue_graph`：线索和推理链；
- `dialogue_rules`：NPC 对话权限和证据回应；
- `choices`、`endings`：关键选择和结局条件；
- `visual_assets`、`visual_style_guide`、`hotspot_positioning`：视觉资产、统一风格和热点坐标。

剧本包不能一次生成后直接入库，至少要经过：

```text
剧本初稿
  -> 编剧审稿
  -> 逻辑 QA
  -> 修复合并
  -> SchemaValidator
  -> ScriptSupervisor
  -> 图片质量门禁
  -> 热点定位校准
  -> ScriptImportService 导入游戏
```

前端展示的生成进度必须来自后端 job `steps`，不能用前端假进度。生成期间可以轮播历史名人名句，但名句只作为过渡氛围，不改变真实进度。

## 7.6 生成剧本视觉质量门禁

生成剧本不能只停留在 prompt 层。场景图、人物图和核心线索图必须真实生成并通过 `ImageQualityGate`，才能进入正式游玩。

必须拒绝并重新请求的图片：

- 空白图、纯色图、损坏图、严重模糊图；
- 占位图、fallback 图、旧缓存图；
- 错朝代建筑、服饰、器物、照明或文字；
- 现代物件、现代字体、现代组织符号；
- 与 `visual_style_guide` 不一致的风格；
- 与 `appearance_lock` 冲突的人物图；
- 集成场景图中缺少核心线索物件；
- 线索物件太小、遮挡或无法定位；
- 热点坐标无法落到真实物件上的图片。

生成剧本必须包含：

- `visual_style_guide`：统一色彩、光照、镜头、材质和负面 prompt；
- `appearance_lock`：锁定每名主要 NPC 的年龄、体态、面部、帽式、服饰和标志物；
- `era_feature_checklist`：每张场景图的朝代建筑、服饰、器物、文书和禁用元素；
- `hotspot_positioning`：核心热点的归一化 `anchor_point` 和 `bbox`。

图片未通过质量门禁时，系统必须改写 prompt 并重新请求生图。必需图片重试后仍不通过时，生成 job 应标记为 `visual_blocked` 或 failed，不得进入正式路演游玩。

# 8. 线索系统设计

## 8.1 线索系统定位

线索系统是本项目的核心玩法，不是装饰功能。

每条线索必须满足至少一个条件：

* 能解锁 NPC 新对话；
* 能推进剧情阶段；
* 能改变 NPC 信任；
* 能影响结局分数；
* 能与另一条线索组合出新判断；
* 能揭示隐藏真相的一部分；
* 能触发 AI NPC 的不同回应。

禁止出现无用线索。

---

## 8.2 线索类型

| 类型     | 示例              | 作用           |
| ------ | --------------- | ------------ |
| 物证     | 残页、血迹、灰烬、路引、账本  | 直接证明事件异常     |
| 证言     | NPC 说法、传闻、矛盾口供  | 用于判断人物可信度    |
| 文书     | 军报、契约、账册、禁书残稿   | 连接制度和时代背景    |
| 环境痕迹   | 锁痕、脚印、火势方向、缺失物品 | 推理事件经过       |
| 身份线索   | 腰牌、印信、称谓、路引     | 判断 NPC 真实身份  |
| 历史知识线索 | 制度规则、禁令、税制、军制   | 防止玩家只靠现代逻辑推理 |
| 反证线索   | 与 NPC 说法矛盾的证据   | 触发质问或反转      |

# 10. AI 接入要求

## 10.1 接入原则

本项目开发中必须真实调用 AI，不能只写 prompt，不测试；不能只用 mock 数据骗演示。

| 要求           | 说明                             |
| ------------ | ------------------------------ |
| API Key 只放后端 | 禁止写入前端代码或 GitHub 公共仓库          |
| 使用 `.env` 管理 | `DEEPSEEK_API_KEY=xxx`         |
| 所有 AI 调用可记录  | 保存 prompt、response、耗时、是否通过监管   |
| 所有 AI 输出结构化  | 优先要求 JSON 输出                   |
| 每个模块都要测试     | 事件生成、NPC 对话、监管器、结局总结、视觉提示词都要实测 |
| 失败必须回退       | API 失败时使用预设文本，不让游戏崩溃           |
| 模型参数可配置      | 模型名、温度、超时、重试次数从配置读取，不硬编码       |

# 11. RAG 与历史知识库设计

## 11.1 RAG 定位

RAG 是幕后能力，不是产品卖点本身。

它的作用是：

> **为 AI 生成世界背景、NPC 台词、线索解释和一致性审查提供历史依据，防止 AI 胡编、错朝代、角色越权。**

---

## 11.2 最终选型

MVP 阶段采用：

> **ChromaDB + SQLite + JSON 文件**

| 组件              | 用途                        |
| --------------- | ------------------------- |
| ChromaDB        | 存储朝代知识、制度规则、历史材料的向量索引     |
| SQLite          | 存储结构化游戏数据：事件、NPC、线索、状态、日志 |
| JSON 文件         | 存储可读、可编辑的初始内容包            |
| 本地 embedding 模型 | 生成知识库向量                   |
| DeepSeek API    | 负责生成、对话、监管，不负责向量库本身       |

# 12. 一致性监管器设计

## 12.1 监管器定位

一致性监管器是技术亮点，也是 AI 叙事可控性的保证。

它负责防止 AI 出现：

* 历史穿帮；
* 错朝代元素；
* NPC 越权；
* 提前剧透；
* 人设漂移；
* 线索冲突；
* 阶段跳跃；
* JSON 格式错误。

---

## 12.2 检查项

| 检查项     | 示例                    |
| ------- | --------------------- |
| 错朝代     | 明代不能出现“互联网”“清代八旗制度”   |
| 身份越权    | 书坊学徒不能命令锦衣卫撤退         |
| 剧透      | 第一阶段不能直接说出幕后主使        |
| 人设漂移    | 胆小刻工不能突然公开指认高官        |
| 线索冲突    | NPC 说法不能无标记地推翻已确认物证   |
| 历史风险    | 火药、毒物等只允许剧情描述，不允许制作细节 |
| 阶段跳跃    | 调查阶段不能直接进入结局          |
| JSON 格式 | 输出必须可解析               |

# 14. 前端 UI 设计

## 14.1 页面布局原则

采用：

> **视觉小说式主舞台 + 推理游戏式线索栏**

背景作为完整舞台铺满主画面，NPC 立绘叠加在背景之上，对话框覆盖底部，线索栏以侧边抽屉或半透明面板形式出现。

前端展示升级为：

> **React 叙事 UI + Phaser 可交互游戏舞台**

Phaser 不替代 React，也不接管剧情逻辑。Phaser 只负责把当前场景可视化成“像游戏一样可点击、会反馈、会转场”的舞台；React 继续负责长文本对话、线索栏、证据出示、主动推理、人物信息、结局页和调试展示；FastAPI 继续作为叙事中枢，负责 AI、RAG、状态机、线索释放、NPC 权限和一致性监管。

一句话边界：

> **Phaser 是舞台，不是大脑；React 是界面，不是剧情裁判；FastAPI 是叙事中枢。**

## 14.5 Phaser 舞台接入目标

Phaser 的接入目标不是“迁移到游戏引擎”，而是将现有 `ScenePanel` 升级为 `PhaserStage`，让 AI 叙事结果在主舞台上形成可感知反馈。

P0 范围：

| 能力       | 要求                                      |
| -------- | --------------------------------------- |
| 背景舞台     | 根据当前 `scene.visual_asset_url` 或 `visual_asset_id` 显示场景背景 |
| NPC 呈现    | 根据 `scene_npcs` 显示 NPC 立绘 / 剪影，并支持当前选中 NPC 高亮 |
| 调查热点     | 根据 `scene.hotspots` 在舞台上生成可点击区域                |
| 热点点击     | 点击后只发出事件，仍由 React 调用 `/api/investigate`        |
| 线索反馈     | 新线索出现时播放高亮、镜头推进、闪光或物件描边                 |
| 氛围特效     | 支持雨、烟、火光、灰烬等轻量粒子或动画效果                   |
| 场景转场     | `scene_id` 变化时播放淡入淡出或暗场转场                  |
| 可回退      | 保留旧 `ScenePanel`，通过前端开关回退到 React 舞台        |

P0 不做：

```text
不做角色行走
不做开放地图
不做复杂小游戏
不做 Phaser 内长文本 UI
不做 Phaser 内 AI 请求
不做 Phaser 内剧情判定
不重构 DialoguePanel / ClueSidebar / EndingPanel
```

## 14.6 React 与 Phaser 通信模型

Phaser 和 React 通过事件桥通信。Phaser 只上报玩家在舞台上的交互，React 继续调用后端并更新 `snapshot`。

```text
玩家点击 Phaser 热点
  ↓
Phaser emit hotspot:clicked
  ↓
PhaserStage 转为 React 回调 onInspect(sceneId, hotspotId, clueId)
  ↓
App.tsx 调用 FastAPI /api/investigate
  ↓
FastAPI 根据状态机、线索规则、NPC 权限和监管器返回结果
  ↓
React 更新 snapshot / actionNotice / 线索栏
  ↓
PhaserStage 将新 snapshot 推送给 MainScene 刷新舞台
```

React 集成要求：

```text
挂载 PhaserStage 时创建 Phaser.Game
卸载 PhaserStage 时 game.destroy(true)
snapshot 更新时发送 snapshot:update 事件
不要每次 React render 都 new Phaser.Game
清理所有 Phaser game.events 监听
```

短期前端可从现有结果推导舞台反馈：

| 条件                 | 舞台反馈                  |
| ------------------ | --------------------- |
| `new_clues.length > 0` | 线索高亮、镜头推进、轻微闪光       |
| `scene_id` 变化       | 淡入淡出 / 暗场转场           |
| `current_stage` 变化  | 镜头推进 / 暗场 / 标题提示       |
| NPC 情绪变化           | NPC 轻微位移、变暗、震动或呼吸变化   |
| 进入结局               | 淡出到结局页                |

P1 再由后端正式返回 `stage_cue`：

```ts
type StageCue = {
  focusHotspotId?: string;
  focusNpcId?: string;
  effects?: Array<'rain' | 'smoke' | 'embers' | 'flash' | 'camera_push' | 'shake'>;
  highlightedClueIds?: string[];
  transition?: 'fade' | 'cut' | 'darken';
};
```

## 14.7 AI 剧本生成前端流程

北宋 / 晚唐入口需要新增生成流程 UI：

```text
朝代选择
  -> 明代：直接进入固定 Demo
  -> 北宋 / 晚唐：关键词输入
  -> 真实生成流程图
  -> 剧本概览
  -> 生成身份选择
  -> start-generated
  -> 游戏主界面
```

关键词输入页要求：

- 只在北宋 / 晚唐显示；
- 关键词 1-8 个；
- 关键词为空时显示中文提示；
- 关键词不能绕过朝代边界；
- 不展示明代固定 Demo 的生成入口。

生成流程图要求：

- 节点状态来自后端 job `steps`；
- 前端不能用定时器假装推进；
- `running`、`passed`、`failed`、`blocked`、`retrying` 要能区分；
- 图片重试、剧本修复、热点重校准必须可见；
- job 失败时停在失败节点并展示中文失败原因。

等待过渡可以轮播历史名人名句，但名句只作为文化氛围，不推动进度条，不暗示某阶段完成。

生成完成后，前端必须展示：

- 来自 `script_overview` 的非剧透剧本概览；
- 来自 `playable_identities` 的可选身份；
- approved 主图或场景缩略图；
- 选择身份后调用 `POST /api/session/start-generated`。

概览和选人页不得展示静态北宋 / 晚唐占位内容、明代书坊案人物、未通过质量门禁的图片、最终真相、隐藏线索或结局条件。

# 16. 主 Demo：明代书坊焚稿案

## 16.1 Demo 定位

主 Demo 是一局完整历史悬疑推理剧本。

目标体验：

> 玩家作为明代书坊学徒，卷入一场看似普通的书坊失火。随着调查推进，玩家发现火灾背后不是单纯禁书，也不是简单官府搜捕，而是一份夹带在诗稿中的边地粮册抄录。书坊掌柜、刻工、落第士子、锦衣卫校尉都只掌握部分真相。玩家必须在保全自己、保护他人、保存证据和接近真相之间做选择。

---

## 16.2 剧本核心

| 项目    | 内容                       |
| ----- | ------------------------ |
| 玩家身份  | 明代书坊学徒                   |
| 表层事件  | 夜间书坊失火，未刊文稿被烧            |
| 表层解释  | 掌柜称灯油翻倒，引发走水             |
| 第一层真相 | 火灾不是意外，而是人为纵火            |
| 第二层真相 | 被烧的不只是诗稿，而是夹带文书抄录的稿件     |
| 第三层真相 | 抄录内容涉及边地粮册与官府往来          |
| 情感核心  | 小人物是否要为一份未必能改变大局的证据冒险    |
| 时代矛盾  | 文字风险、书坊生计、官府侦缉、边地财政、士人名节 |
| 核心问题  | 真相是否值得用个人命运去交换？          |

---

## 16.3 五阶段剧情

### stage_1_intro：夜火

目标：

* 让玩家知道书坊失火；
* 掌柜给出“灯油走水”的表层解释；
* 锦衣卫校尉陆峥封锁现场；
* 玩家发现第一批异常线索。

可释放线索：

| 线索     | 作用          |
| ------ | ----------- |
| 火起点异常  | 证明火灾不像灯油架引起 |
| 烧焦残页   | 出现“辽东粮册”四字  |
| 半枚红印纸角 | 暗示可能不是普通诗稿  |

---

### stage_2_investigation：各执一词

NPC 表层说法：

| NPC | 表层说法                  |
| --- | --------------------- |
| 许掌柜 | “灯油翻了，烧掉的是几卷无关紧要的旧稿。” |
| 阿沈  | “我整夜在刻坊，什么都没听见。”      |
| 顾闻  | “我的诗稿被烧了，我只是来讨个说法。”   |
| 陆峥  | “奉命封锁，查明之前无人离开。”      |

可释放线索：

| 线索       | 来源      | 作用          |
| -------- | ------- | ----------- |
| 异常火油味    | 后院火场    | 证明有人助燃      |
| 刻工矛盾证言   | 刻坊对话    | 阿沈并非整夜无知    |
| 掌柜提前清空书箱 | 前厅 / 刻坊 | 掌柜隐瞒失火前动作   |
| 士子焦急寻找旧稿 | 顾闻对话    | 顾闻在意的不是普通诗稿 |

---

### stage_3_reversal：烧的不是书

反转逻辑：

```text
玩家原以为：
书坊烧的是犯禁文稿。

后来发现：
犯禁只是表层风险。
真正被销毁的是夹带在诗稿中的边地粮册抄录。

进一步发现：
锦衣卫校尉陆峥并不知道完整内容，他只是被上级命令封锁书坊。
```

关键线索：

| 线索     | 来源    | 作用           |
| ------ | ----- | ------------ |
| 士子夹带抄录 | 雨巷私谈  | 证明诗稿中夹有其他文书  |
| 锦衣卫封口令 | 临时审问处 | 说明官府只想压下事件   |
| 城门搜检加严 | 城门消息  | 说明事件已扩散到书坊之外 |
| 陆峥命令矛盾 | 陆峥对话  | 说明陆峥也被利用     |

---

### stage_4_choice：证据与活路

| 行动       | 倾向      | 后果                  |
| -------- | ------- | ------------------- |
| 将残页交给陆峥  | 秩序      | 陆峥可能保护玩家，但证据可能被上级截断 |
| 暗中交给顾闻   | 真相 / 牺牲 | 证据有机会流出，但顾闻风险极高     |
| 帮掌柜销毁余稿  | 自保      | 书坊暂时保全，但真相断裂        |
| 用线索逼阿沈作证 | 真相      | 需要阿沈信任，否则他会崩溃或反咬    |
| 反查陆峥上级命令 | 隐藏      | 风险最高，但可能触发隐藏结局      |

---

### stage_5_ending：历史回声

每个结局文本必须包含：

* 玩家关键选择；
* 玩家发现或忽略的关键线索；
* 至少 2 名 NPC 的命运；
* 事件表层结果；
* 更大的历史结构没有被完全改变；
* 一句“历史回声”式总结。

---

## 16.4 主要 NPC

| NPC    | 身份      | 表面目标        | 隐藏动机              | 玩家对其误判                |
| ------ | ------- | ----------- | ----------------- | --------------------- |
| 许掌柜    | 书坊掌柜    | 说服众人相信是意外失火 | 保住书坊，也保住自己提前搬箱的秘密 | 玩家会以为他是主谋             |
| 阿沈     | 刻工      | 证明自己与火灾无关   | 听见三更搬箱，但害怕被牵连     | 玩家会以为他胆小无用，实际掌握关键时间线  |
| 顾闻     | 落第士子    | 找回自己的诗稿     | 诗稿中夹带过一份粮册抄录      | 玩家会以为他只是文人清高          |
| 陆峥     | 锦衣卫低级校尉 | 封锁书坊，执行命令   | 逐渐发现上级命令不完整       | 玩家会以为他是压迫者，后续可发现他也被利用 |
| 未露面的上级 | 幕后压力源   | 不直接登场       | 要求压下文稿来源          | 通过命令、封口令和城门搜检体现       |

---

## 16.5 多结局

| 结局        | 条件                         | 结果                 | 历史回声                   |
| --------- | -------------------------- | ------------------ | ---------------------- |
| 自保结局：灰烬无声 | survival ≥ 4，玩家销毁或隐瞒关键残页   | 书坊暂时保全，顾闻失踪，阿沈继续沉默 | 你保住了自己，却让那几行字彻底变成灰     |
| 秩序结局：交还官府 | order ≥ 4，玩家将证据交给陆峥        | 陆峥带走残页，事件被定为普通失火   | 你选择相信秩序，但秩序未必选择真相      |
| 真相结局：残页出城 | truth ≥ 5，关键线索齐全，顾闻或阿沈信任达标 | 证据被带出城，书坊被封，玩家被迫离开 | 一页残纸活了下来，但没有人知道它能活多久   |
| 悲剧结局：封口   | risk_level ≥ 6 且 NPC 信任不足  | 玩家被牵连，阿沈反口，顾闻被捕    | 你接近了真相，却没有足够的人愿意与你站在一起 |
| 隐藏结局：借刀留痕 | 找齐隐藏线索，陆峥信任 ≥ 2，顾闻信任 ≥ 2   | 陆峥表面结案，暗中放走顾闻，保留副本 | 你没有推翻黑幕，只是在它合拢前留下了一道缝  |

# 22. 当前最关键开发决策

最终建议：

**第一，明代继续作为稳定固定 Demo。**
明代书坊焚稿案不接入 P0 AI 剧本生成，继续承担路演主流程、fallback 和回归测试职责。

**第二，北宋和晚唐承接 AI 剧本生成。**
北宋 / 晚唐不再只是静态预览。用户选择这些朝代后先填写关键词，再进入 DeepSeek 生成、打磨、图片门禁、热点定位、概览选人和生成剧本游玩流程。

**第三，视觉系统升为 P0，且图片质量门禁是阻塞验收。**
必须有主视觉、场景背景、NPC 概念图和关键线索图，并通过 Phaser 舞台呈现场景、热点、NPC、火光雨雾和线索高亮，否则不够贴“剧情驱动类”的赛题表达。
生成剧本中，空白图、占位图、fallback 图、旧缓存图、错风格图、错朝代图或缺少核心线索物件的图片不能计入完成，必须重试或标记 `visual_blocked`。

**第四，RAG 放到幕后。**
RAG 是可信生成的支撑，不是第一卖点。对外讲“AI 构建历史事件世界”，对内用 RAG 约束历史可信。

**第五，线索系统必须优先开发。**
没有线索栏、红色高亮、线索详情和线索组合，这个项目会退化成普通 AI 对话游戏。

**第六，所有关键剧情不能交给 AI 自由决定。**
AI 负责剧本草拟、世界细化、NPC 台词、动作、情绪、视觉提示词和结局文本；线索释放、阶段跳转、结局条件、图片通过与否、热点坐标通过与否必须由代码和规则控制。

**第七，Phaser 只增强 `ScenePanel` 舞台层。**
当前阶段不做全局前端重构，不移动 `App.tsx` 主状态，不替换 `DialoguePanel`、`ClueSidebar` 和 `EndingPanel`。Phaser 的 P0 目标是把现有视觉小说主舞台升级为可点击、会反馈、会转场的游戏舞台。

# 23. 作品简介新版

> 《史隙》是一款 AI 驱动的历史悬疑分支叙事游戏。玩家可进入明代、北宋、晚唐等历史时期：选择明代时直接进入稳定的书坊焚稿案固定 Demo；选择北宋或晚唐时，玩家填写关键词，由 DeepSeek 多轮生成和打磨一份完整可玩的历史悬疑 ScriptPackage，包括故事、人物、地点、线索链、对话规则、关键选择、结局和视觉资产需求。系统会基于朝代知识库和事件语法库生成本局世界背景、人物关系网、关键场景与核心冲突，并由 AI NPC 根据玩家推理进度、已发现线索、信任度和自身知识权限动态回应。前端采用 React + Phaser 混合叙事舞台：React 负责对话、线索栏、证据出示、推理、生成流程图、剧本概览、选人和结局展示；Phaser 负责场景背景、NPC 入场、可点击调查热点、线索高亮、火光雨雾和转场反馈。项目同时通过 AI 生成主视觉、场景图、NPC 图和线索图，并用图片质量门禁拒绝占位图、空白图、错风格图和错朝代图，用热点定位校准确保线索物件可交互。为避免 AI 叙事常见的历史穿帮、角色越权、提前剧透和逻辑跳跃，系统设计了一致性监管器和剧本监管器，对 AI 输出进行历史规则、人物权限、剧情状态、线索链、结局可达性和视觉可用性审查，最终实现一个可重玩、强故事线、可监管、多结局的 AI 剧情驱动游戏原型。

# 24. 最终判断

这个 V3.1 版本的核心定位是：

> **不是“AI 历史知识检索系统”，而是“AI 剧情驱动历史悬疑游戏”。**

真正亮点有九个：

第一，**明代稳定 Demo 与北宋 / 晚唐 AI 生成并存**。
明代保持可演示、可回归、可 fallback 的固定主线；北宋和晚唐用关键词触发 DeepSeek 生成新剧本，既稳又能展示扩展能力。

第二，**AI 构建本局历史事件世界**。
它不只是检索资料，而是生成地点、时代压力、事件前史、NPC 关系和当前冲突。

第三，**AI 生成完整可玩的 ScriptPackage**。
剧本包不只是故事梗概，而包含 `script_overview`、`playable_identities`、地点、NPC、线索图、对话规则、选择、结局、视觉资产和热点定位。

第四，**AI 生成关键视觉与概念美术**。
主视觉、场景图、NPC 图、线索图都纳入 P0，贴合剧情驱动类赛题要求。

第五，**AI NPC 根据推理进度推进剧情**。
NPC 不是聊天工具，而是受角色卡、知识权限、信任度和线索状态控制的剧情推动者。

第六，**线索系统有实际玩法价值**。
红色高亮、线索栏、线索详情、出示线索、线索组合让玩家真正参与推理。

第七，**分支叙事与多结局有代价**。
结局不是好坏二分，而是围绕真相、秩序、自保、牺牲和历史惯性展开。

第八，**AI 被监管而不是放任**。
RAG、状态机、NPC 权限、一致性监管器、剧本监管器、图片质量门禁和热点定位校准共同限制 AI，避免剧情、视觉和交互失控。

第九，**Phaser 把 AI 叙事结果变成游戏舞台反馈**。
AI 和状态机返回的场景、人物、线索和阶段变化，会被 Phaser 表现为可点击热点、NPC 站位、线索高亮、火光烟雾、镜头推进和场景转场，让作品从“AI 文本交互”升级为“AI 驱动网页游戏”。

这版更适合拿去做黑客松申报、开发拆解和路演讲解。
