# 01 MVP 范围

## MVP 原则

MVP 只服务一个目标：完整跑通“明代 / 书坊学徒 / 书坊焚稿案”的可演示闭环。任何不能帮助这局 Demo 跑通、解释 AI 价值、展示线索玩法或证明监管能力的功能，都推迟。

## 明确数量范围

| 项目 | MVP 数量 | 说明 |
| --- | ---: | --- |
| 朝代数量 | 1 个完整朝代 + 2 个占位入口 | 明代完整；北宋、晚唐只保留禁用/预告入口 |
| 身份数量 | 1 个可玩身份 + 2 个占位身份 | 明代书坊学徒可玩；其他身份后置 |
| 主事件数量 | 1 个 | 明代书坊焚稿案 |
| NPC 数量 | 4 个核心 NPC + 1 个幕后压力源 | 许掌柜、阿沈、顾闻、陆峥；幕后上级不直接对话 |
| 场景数量 | 5 个 | 前厅、后院火场、刻坊、雨巷、临时审问处 |
| 线索数量 | 15 条 | 至少 8 条关键线索，7 条辅助/反证线索 |
| 结局数量 | 5 个 | 自保、秩序、真相、悲剧、隐藏 |
| AI 调用点数量 | 最少 3 类 | NPC 对话、监管器、历史回声；视觉提示词可第 2 批接入 |
| 真实 API | Mock 之后必须接入 | 先跑 Mock，再接 DeepSeek 或兼容模型 |
| RAG | MVP 可先关键词检索 | 向量数据库后接；接口先预留 |
| 图片生成 | 非首个可运行版本必需 | 首版可占位图；路演版需视觉提示词和部分图片 |

## P0 功能

P0 是没有它就不算 Demo 的功能：

- React 游戏主界面。
- FastAPI 后端基础服务。
- 明代书坊焚稿案固定数据包。
- 会话创建和读取。
- 五阶段状态机。
- 场景调查。
- 红色高亮线索点击。
- 线索栏与线索详情。
- NPC 对话和出示线索。
- Mock NPC 回复。
- 真实 AI NPC 对话开关。
- 一致性监管器最小规则。
- 结局判定。
- AI 调用日志。
- 基础测试。

## P1 功能

P1 是路演体验增强功能：

- 视觉小说 UI 打磨。
- NPC 剪影 / 占位立绘。
- 场景背景图或 AI 生图结果。
- 视觉提示词生成。
- RAG 关键词检索增强。
- AI 修复 Agent。
- 调试面板展示状态机、线索、AI 日志。
- 北宋、晚唐入口展示但不可完整游玩。

## P2 功能

P2 是后续扩展，黑客松期间不要优先做：

- 完整北宋、晚唐事件。
- 多身份完整路线。
- 多事件模板自动生成。
- 复杂推理图谱编辑器。
- 用户账户系统。
- 云端多人存档。
- 复杂向量库后台管理。
- 自动美术资产批量生成工作流。

## 当前阶段必须砍掉的功能

- 不能做全朝代开放选择。
- 不能做无限自由输入驱动所有剧情。
- 不能让 AI 自由决定真相和结局。
- 不能做复杂地图、背包、战斗、养成。
- 不能做大型内容管理后台。
- 不能先做部署而不做可玩闭环。

## 为什么先做明代书坊焚稿案

明代书坊焚稿案具备最适合 MVP 的条件：

1. 空间小，5 个场景即可形成闭环。
2. 人物少，4 个 NPC 足以产生冲突。
3. 线索自然，文稿、火场、印纸、封口令都能做成可点击线索。
4. 历史氛围明确，书坊、文稿风险、侦缉压力容易被玩家理解。
5. 结局选择有代价，适合展示“真相、秩序、自保”的分支。

## 一局完整 Demo 的最低内容量

最小完整局必须包含：

- 1 个开场剧情。
- 5 个可进入场景。
- 4 个可对话 NPC。
- 15 条可收集线索。
- 3 组线索组合。
- 5 个剧情阶段。
- 5 个结局。
- 至少 6 次 NPC 回复。
- 至少 1 次监管器拦截展示。
- 至少 1 个 AI 调用日志展示。

## 最小可运行版本完成标准

满足以下条件才算可运行：

1. `npm run dev` 可打开前端页面。
2. 后端健康检查返回正常。
3. 前端可创建新会话。
4. 玩家能从 intro 走到 ending。
5. 至少收集 8 条线索。
6. 至少和 2 个 NPC 完成对话。
7. 至少一次出示线索改变 NPC 回复。
8. 不接真实 AI 时 Mock 仍可完整通关。
9. 接真实 AI 失败时自动回退 Mock 文本。
