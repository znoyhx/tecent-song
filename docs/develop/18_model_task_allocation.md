# 18 模型任务分配

## 模型等级定义

- 强模型：负责架构、状态机、AI 链路、监管器、复杂审查和最终整合。
- 普通模型：负责清晰边界内的服务、组件、测试和数据实现。
- 低级模型：只负责单文件或小范围任务，不允许自行设计架构。

## 强模型适合做什么

- 架构设计和目录规划。
- API 合约审查。
- 数据模型审查。
- 状态机和结局逻辑。
- 一致性监管器。
- AI prompt 和修复策略。
- 合并前代码审查。

## 普通模型适合做什么

- 实现明确 API 路由。
- 实现数据加载服务。
- 实现前端组件。
- 编写单元测试。
- 编写 Mock 数据。

## 低级模型适合做什么

- 按模板新增 JSON 数据。
- 实现简单展示组件。
- 修复样式问题。
- 补充测试用例。
- 小范围 bug 修复。

## 禁止交给低级模型的任务

- 修改状态机核心规则。
- 修改 API 合约字段。
- 修改 AIClient 密钥处理。
- 修改监管器判断逻辑。
- 大规模重构目录。
- 删除 Mock Demo。
- 修改 `docs/PRD.md` 和本执行文档。

## 降低模型成本

- 先让低级/普通模型实现 Mock 和 UI。
- 强模型只审查关键模块和集成点。
- Prompt 中明确文件范围，避免全仓扫描。
- 每次任务不超过 3-5 个文件。
- 先写测试再让低级模型实现。

## 防止低级模型乱改代码

每个任务必须写明：

- 只改哪些文件。
- 禁止改哪些文件。
- 输入和输出。
- 验收命令。
- 不允许重构目录。
- 完成后列出改动文件。
